GRIT Core Committee Runbook

This page explains what the core committee does to make a cycle live, route ideas to jury panels, and close the competition safely.

Go Live Flow

User Screen Overview

The user portal is a competition workspace: entrants browse visible ideas by category tab with pagination, open only the detail they need, submit during the active cycle, preview rich/plain content, and inspect uploaded images in a larger preview.

Core Checklist

  1. Start the app with python run.py and log in to the core portal.
  2. Choose the six-month Cycle 1 or Cycle 2 slot, year, start date/time, and expiry date/time. The app builds GRIT-Cycle1-YYYY or GRIT-Cycle2-YYYY; the user portal only accepts new submissions during that window and its countdown updates live. This one cycle window applies to every category in the cycle.
  3. Review categories, winner target counts, and category names before entrants submit.
  4. Create core, jury lead, and jury member logins in /core/users.
  5. Add protected usernames to app/access_config.py: core gets {"core"}; jury lead and jury member accounts get {"jury"}.
  6. Assign jury leads and jury members in /core/categories. You can create a new jury account directly from the category panel, remove old panel members, and save replacements. Each active jury panel must contain three to seven unique people.
  7. After submissions close or the committee is ready, release the cycle to jury from the core dashboard. During pre-production testing, use Withdraw release to block jury scoring again.
  8. After jury lead confirmation, open Final Winners to review grouped category winners with scores and lead final comments, then close jury visibility and archive the cycle.

Account Removal

Disable the login in /core/users, remove the username from app/access_config.py, and uncheck it from category panel assignments when it should no longer review that cycle.

Credential Handling

Portal accounts are stored in MongoDB collection users. Passwords are hashed before storage. Do not grant access by putting usernames in URLs; protected views require login, the username access map, and category assignment for jury users. The seeded diptanun account is an intentional test account with both core and jury access.

For category-specific jury sign-in, create separate jury lead and jury member accounts per category with distinct passwords, add those usernames to app/access_config.py, then assign them only to the category panel they should review.

SQLite Demo Mode

The local approval demo uses data/grit.sqlite3 when config.ini has sqlite = yes. The same logical collections are retained as JSON documents and survive restart. Use sync_mongo_to_sqlite.py to copy Mongo data into SQLite, and sync_sqlite_to_mongo.py to promote approved SQLite data back to Mongo later.

Archive And Edit Rules

Archiving marks the cycle and ideas as archived, removes them from active user/core/jury dashboards, and keeps them visible in core archive. After the due date, users cannot submit new ideas but can edit an existing idea with the edit token until core releases to jury. After jury release, idea edits and category changes are locked.

Integrity Note

A person may submit ideas through the public entrant portal and also have a protected core or jury portal account. Jury self-scoring is blocked when the logged-in jury username appears as a contributor username on the idea. Keep ordinary core and jury protected usernames separate unless governance explicitly approves an overlap.

Production Hardening

Before broad launch, add enterprise authentication, CSRF protection, audit event writes, stronger upload controls, and notification delivery if the committee wants automated handoffs.
