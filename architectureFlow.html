Role-Based Flow With Hand-Offs

Users
Browse portalOpen /users/, search ideas, filter by category, view likes/comments.
Submit ideaSubmit problem, solution, Mumbai/Bangalore office, country India, contributors, sponsor, category, images, and details.
Edit before releaseNew submissions close at due date. Existing ideas can be edited with the private token until core releases to jury.
Core Committee
Create cycleSet GRIT-Cycle1-YYYY or GRIT-Cycle2-YYYY, one start/end date for all categories.
Configure categoriesAdd/remove categories, set winner target per category, and manage core members.
Assign panelsKeep exactly 1 lead and 3-5 members per category. Warnings flag missing leads or invalid panel sizes.
Release to juryRelease locks idea/category edits. Withdraw release remains available for testing.
Jury Members
Access assigned categoryJury accounts see only categories where their user ID is assigned.
Score every ideaGive 1-10 score, comment, and signal. Scored rows change color; pending counts remain visible.
Update after discussionScores can be changed while the jury review window remains open.
Jury Lead
Review averagesLead sees category averages, member scores, member comments, and pending completion.
Refresh scoresRefresh pulls latest member updates after panel discussion.
Confirm winnersEnabled only after every assigned reviewer scores every released idea in the category.
Core handoffConfirmed winners appear in core final winners, sorted by score with lead comments.

cyclesOne document per GRIT cycle. Stores name, start/end dates, jury release state, jury close state, and archive state.
categoriesOne document per category per cycle. Stores category name, winner target, active flag, assigned jury lead IDs, assigned member IDs, and confirmed winner IDs.
ideasEach idea stores its cycle ID, selected category IDs, submitter, contributors, content, images, comments, reactions, edit token, submitted date, and last edited date.
evaluationsOne score/comment/signal per idea, category, and juror. This lets the same idea be evaluated separately when it belongs to two categories.
usersCore, jury lead, and jury member accounts with hashed passwords and active/disabled state.
idea_reactionsPer-visitor public like/neutral/dislike events, while cached counts stay on the idea for fast list pages.

Cycle-wise data shape

cycle._id -> categories.cycle_id cycle._id -> ideas.cycle_id category._id -> ideas.category_ids[] category._id -> evaluations.category_id user._id -> categories.jury_lead_ids[] / jury_member_ids[] idea.idea_id -> evaluations.idea_id / idea_reactions.idea_id

Mongo mode stores these as collections in the configured MongoDB database. SQLite demo mode stores the same logical documents as JSON rows in data/grit.sqlite3. Flip sqlite = no in config.ini to use MongoDB again.

Default demo DBSQLite is the default local demo backend. Data persists after app restart in data/grit.sqlite3.
Mongo parityRoutes use the shared storage API, so the same functional changes apply to SQLite and MongoDB.
Sync scriptssync_mongo_to_sqlite.py copies Mongo data to local demo DB. sync_sqlite_to_mongo.py promotes approved demo data to Mongo.
Archive visibilityArchived cycles disappear from active user, core, jury, and lead dashboards, while remaining in core archive.

Operational notes

What To Say In A Demo

Start simple.

Show that users can submit ideas without login, while core and jury have protected portals.

Governance is cycle-based.

Every cycle has one submission window and one jury release state. Categories inherit that cycle window.

Evaluation is category-scoped.

Jury members see only assigned categories. Jury leads see same-category peer scores and comments.

Final winners are controlled.

Core chooses winner target per category. Jury lead can confirm only after complete scoring and a confirmation prompt.
