Public Users

Submitting And Viewing Ideas

No login requiredUsers can browse, search, view, react, comment, and submit ideas without a password.
Cycle awareSubmissions are accepted only during the cycle start/expiry window set by core committee. All categories in a cycle share the same deadline.
Edit protectionOnly the submitter with the private edit token/session can edit an idea. Edits remain open after due date until core releases to jury.

User Features

  • Browse all submitted ideas in a compact paginated gallery.
  • See submitted date and last edited date on idea detail.
  • Filter by category tabs and search by idea ID, problem, solution, owner name, employee ID, or team name.
  • Submit problem statement, solution, optional video link, patent flags, deployed-on-PROD status, contributors, VP-and-above sponsor, office location Mumbai/Bangalore, country India, employee ID, and categories.
  • Add rich HTML or plain-text solution details and preview before submitting.
  • Upload images and optionally provide image display names.
  • Open image previews from the idea detail page.
  • Add public feedback comments with like/neutral/dislike signal.
  • React directly with like/neutral/dislike buttons. Counts show on both gallery and detail pages.

Core Committee

Managing The Competition

Core Features

  • Create and edit cycle windows using GRIT-Cycle1-YYYY and GRIT-Cycle2-YYYY.
  • Control the submission start and end date in one place: /core/cycles. All categories under that cycle use the same start and deadline.
  • View total submitted ideas, category counts, patent counts, location counts, and management dashboard summaries.
  • Filter dashboard and archive ideas by search text, patent status, and Mumbai/Bangalore office location.
  • Use the governance watchlist to discuss next useful views: exports, reviewer SLA, patent triage, and audit trail.
  • Add, edit, disable, and change passwords for core/jury portal accounts.
  • Create and assign a new jury lead/member directly from the Categories page.
  • Add an existing jury lead/member to any category from the Categories page using the Add selected control.
  • Add or disable core committee members from the Categories page or Accounts page; the number of core members is variable.
  • Add, rename, deactivate, and configure categories.
  • Set the winner target per category.
  • Assign exactly one jury lead and three to five jury members per category. The Categories page warns when a panel is missing a lead or has too few/many members.
  • Remove a category lead/member by clicking the red remove icon beside the assigned person and confirming twice.
  • Testing Release ideas to jury and withdraw release before production launch.
  • Close jury visibility after review.
  • Open Final Winners to see jury-lead-confirmed winners grouped by category, sorted by score descending, with lead final comments.
  • Archive a completed cycle so old ideas move out of the active UI.
  • View core-only Swagger and OpenAPI definitions.

Jury Lead

Leading Category Review

  • Can access only categories assigned by core committee.
  • Can score and comment on ideas in the assigned category after core releases to jury.
  • Can see same-category average scores and peer review counts.
  • Can see same-category jury member comments/scores from the category dashboard and idea detail.
  • Can refresh scores after jury discussion when members update their ratings.
  • Can add the final overall winner reference comment.
  • Can confirm top ideas for that category based on the core-set winner target only after all assigned reviewers score every idea.
  • Receives an are-you-sure confirmation prompt before finalizing winners.
  • Cannot see or score other categories unless assigned to them.

Jury Members

Scoring Assigned Ideas

  • Can access only assigned categories after core releases to jury.
  • Can score each idea from 1 to 10.
  • Can add comments and like/neutral/dislike signal for review context.
  • Can see own scored and pending counts.
  • Already-scored ideas show a scored color; pending ideas show a pending color.
  • Idea titles and table rows are kept compact so long descriptions do not stretch the dashboard.
  • Cannot see peer scores or category averages unless they are the assigned jury lead.
  • Cannot score an idea where their username is listed as a contributor.

Typical Testing Flow

End-To-End Walkthrough

  1. Start the app with python start_grit.py.
  2. Open the user portal and submit or inspect ideas.
  3. Log in as core using core.demo / GirtDemo123!.
  4. Open Cycles and verify the start/expiry window.
  5. Remember that this one cycle window applies to every category in the cycle.
  6. Open Categories and confirm each category has exactly one jury lead and three to five jury members.
  7. Add a jury lead/member from Categories with Add existing jury lead/member or Create new jury account, and remove one with the red remove icon beside the assigned name.
  8. Add or disable core committee members directly from Categories when committee membership changes.
  9. Open Core dashboard and click Release to jury.
  10. Log in as a jury member and score assigned ideas.
  11. Log in as the jury lead to see same-category scores and confirm top ideas.
  12. Log back in as core and open Final Winners to review grouped category winners with scores and lead comments.
  13. For testing, core can click Withdraw release to block jury scoring again.
  14. After final review, core can close jury visibility and archive the cycle.

Data And Security Notes

Current Behavior

  • Portal passwords are stored as hashes in MongoDB, not plaintext.
  • In SQLite demo mode, the same logical collections are persisted as JSON documents inside data/grit.sqlite3.
  • Core can reset individual passwords in Accounts and category-level jury passwords in Categories.
  • Jury leads and jury members can click Forgot password; the request is stored for core review without revealing account existence on the login screen.
  • Category jury members can share one password, while the category jury lead has a separate password. Only hashes are stored.
  • Public users do not need accounts.
  • Protected URL visibility is maintained in app/access_config.py.
  • Core/jury accounts are stored in MongoDB collection users.
  • Ideas are stored in ideas; jury scores are stored in evaluations; public reactions are stored in idea_reactions.
  • Cached reaction counts are stored on ideas so gallery pages load quickly.
  • Jury scoring is blocked until core releases to jury.
  • Jury release can be withdrawn for pre-production testing.
  • Category documents do not store separate deadlines. They inherit visibility from their parent cycle.
  • After due date, users cannot submit new ideas but can edit existing ideas until release to jury.
  • After release to jury, idea category edits are locked.
  • Archived cycle ideas are hidden from active user/core/jury dashboards and retained in core archive.
