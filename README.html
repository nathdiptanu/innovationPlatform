GRIT Architecture

GRIT is a Flask 3 application for Python 3.11.1 with MongoDB persistence, OpenAPI definitions, Swagger UI, public idea submission, core committee management, and jury evaluation.

For approval demos, the platform can run fully on local SQLite by setting sqlite = yes in config.ini. The local database is data/grit.sqlite3.

MongoDB

Collections: users, cycles, categories, ideas, evaluations, and reserved audit_events. In SQLite mode, these are JSON documents in data/grit.sqlite3. In Mongo mode, the setup script creates collections, builds indexes, and bootstraps the first core user from environment variables.

Entitlements

Username-to-screen visibility is maintained in app/access_config.py. Core usernames receive the core screen only, jury usernames receive the jury screen only, and category-level jury assignments are stored on category documents. The public user portal remains available to people who also have protected portal accounts. Runtime checks live in app/entitlements.py.

Production Hardening

The pilot flow is covered. Before broad launch, add enterprise authentication, CSRF protection, audit logging, hardened upload handling, and notification delivery if committee handoffs need email or Teams messaging.

Credentials and Data

Portal usernames live in the MongoDB users collection. Passwords are stored as Werkzeug hashes, not plaintext. Core operators add or disable logins in /core/users, reset individual passwords in Accounts, and reset category lead/member passwords in /core/categories. Jury lead/member category assignment is managed in /core/categories. The seeded diptanun account is an explicit test overlap with both core and jury visibility.

Jury leads and jury members can use /auth/forgot-password. The app stores a generic reset request for core review without exposing whether the username exists. Category jury members can share one password, while the category jury lead has a separate password; only hashes are stored.

Seeded jury accounts are category-specific. For example, Automation lead uses jury.lead.automation and a separate Automation password, while Process Improvement has its own lead/member usernames and passwords. Each category is seeded with exactly one lead and four members. Category funneling is enforced by category jury assignments, and only the lead assignment can confirm winners.

The public countdown updates every second while entries are open. Core controls the active window from /core/cycles by selecting Cycle 1 or Cycle 2, the year, start time, and expiry time for each six-month GRIT cycle.

Archiving removes submitted ideas from active user/core/jury screens and keeps them in the core archive. After due date, users cannot submit new ideas but can edit existing ideas until release to jury; after release, edits and category changes are locked.

Demo idea IDs use the cleaner format GRIT-Cycle1-2026-045. Core management dashboard has a countdown, patent/location filters, governance watchlist, and each winner target links back to the category settings screen.

start_grit.py preserves persisted data on restart. It seeds demo records only when the ideas collection is empty, so user submissions and review data are not reset on every server start.
